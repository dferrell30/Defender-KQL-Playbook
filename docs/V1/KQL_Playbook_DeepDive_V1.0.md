KQL Playbook Deep Dive V1.0

⸻

🧭 Investigation Model

This playbook is built around a simple investigation model:

Hunt → Pivot → Investigate → Validate → Decide

Instead of asking:
“What queries should I run?”

The focus becomes:
“What is this behavior actually telling me?”

Each query should lead to a question.
Each question should lead to a pivot.
Each pivot should lead to clarity.

⸻

🔁 Entity Pivoting Strategy

Investigation is not table-based. It is entity-based.

Primary pivot points:

* User (AccountUpn / AccountName)
* Device (DeviceName / DeviceId)
* IP Address
* URL / Domain
* Email Message ID

These allow movement across:

* Email (Defender for Office 365)
* Identity (Entra ID / Conditional Access)
* Endpoint (Defender for Endpoint)

The goal is to follow activity—not stay in one dataset.

⸻

🔗 Cross-Layer Correlation

Attacks do not stay in one layer.

Typical flow:

Email → Identity → Endpoint

KQL allows correlation across layers using shared entities.

Example:

EmailEvents
| where ThreatTypes has “Phish”
| join IdentityLogonEvents on AccountUpn
| project Timestamp, AccountUpn, IPAddress

🔍 Hunting & Investigation Queries

⸻

📧 Suspicious Email Activity

What this is doing

Identifies phishing or suspicious email delivery.

Query

EmailEvents
| where ThreatTypes has “Phish”
| project Timestamp, SenderFromAddress, RecipientEmailAddress, Subject, ThreatTypes

Pivot

* UrlClickEvents
* IdentityLogonEvents

What to look for

* repeated targeting
* suspicious sender domains
* multiple recipients impacted

⸻

🔗 URL Click Activity

What this is doing

Determines whether a user interacted with a potentially malicious link.

Query

UrlClickEvents
| project Timestamp, AccountUpn, Url, ActionType

Pivot

* IdentityLogonEvents
* DeviceEvents

What to look for

* successful clicks
* unfamiliar domains
* correlation with login activity

⸻

🔐 Identity Sign-In Activity

What this is doing

Analyzes user authentication behavior.

Query

IdentityLogonEvents
| project Timestamp, AccountUpn, IPAddress, Location, ResultType

Pivot

* DeviceProcessEvents
* additional login activity

What to look for

* impossible travel
* unusual IP addresses
* login success after failures

⸻

💻 Endpoint Process Activity

What this is doing

Identifies suspicious processes executed on endpoints.

Query

DeviceProcessEvents
| project Timestamp, DeviceName, AccountName, FileName, ProcessCommandLine

Pivot

* DeviceNetworkEvents
* AlertInfo

What to look for

* unusual process names
* suspicious command lines
* unexpected execution paths

⸻

🌐 Endpoint Network Activity

What this is doing

Analyzes outbound connections from devices.

Query

DeviceNetworkEvents
| project Timestamp, DeviceName, RemoteIP, RemoteUrl

Pivot

* DeviceProcessEvents
* IdentityLogonEvents

What to look for

* connections to unknown domains
* repeated external communication
* correlation with suspicious processes

⸻

🚨 Alerts & Evidence

What this is doing

Surfaces alerts generated by Defender.

Query

AlertInfo
| project Timestamp, Title, Severity, Category

Pivot

* AlertEvidence
* related device/user activity

What to look for

* high severity alerts
* repeated alert patterns
* correlation with other activity

⸻

🧪 Validation Layer

Deployment does not guarantee visibility.

Validation confirms what actually happened.

Key questions:

* Was the activity captured?
* Did expected signals appear?
* Were alerts triggered?
* Is anything missing?

Security tools often fail quietly.

Validation is how you detect those gaps.

⸻

🎯 From Hunting to Detection

Hunting is not the end state.

When behavior becomes:

* repeatable
* reliable
* low noise

It should be turned into detection.

This allows:

* continuous monitoring
* faster response
* reduced manual investigation

Hunting finds behavior.
Detection operationalizes it.
